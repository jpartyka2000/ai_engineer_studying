**How a commit's hash is determined**

A Git commit hash (like `bddff1e`) is a SHA-1 checksum computed from several pieces of data:

- **The content** — the actual file changes (the "tree" object)
- **The parent commit(s)** — the hash of the previous commit(s)
- **The author** — name, email, and timestamp
- **The committer** — name, email, and timestamp (can differ from author in rebases or cherry-picks)
- **The commit message**

**Why this matters for rebasing:**

If you change *any* of these, you get a different hash. During a rebase:

- The content (your changes) stays the same
- The commit message stays the same
- The author stays the same
- But the **parent commit changes**

That's enough to produce a completely different hash.

**Example:**

Your original commit `D` might have:
- Parent: `B` (hash: `abc123`)
- Content: "add login feature"
- Result: hash `def456`

After rebasing onto `C`, the new commit `D'` has:
- Parent: `C` (hash: `xyz789`)
- Content: "add login feature" (identical)
- Result: hash `fff111` (completely different)

**This is also why amending a commit creates a new hash** — even changing just the commit message produces a new commit, because the message is part of the hash calculation.
