Skill Level: Medium
Category: Branching & Merging
Type: Scenario / Multiple Choice
------------------------------------------------------------
QUESTION:
You're merging a feature branch into `main` and Git reports 
a merge conflict in `app.js`. You see:

function getUserName() {
<<<<<<< HEAD
    return this.firstName + " " + this.lastName;
=======
    return `${this.firstName} ${this.lastName}`;
>>>>>>> feature-branch
}

Which statement is TRUE?

A) Code between <<<<<<< HEAD and ======= is from feature-branch
B) Code between ======= and >>>>>>> is from main
C) Code between <<<<<<< HEAD and ======= is from main
D) Conflict markers should be left in after resolving
------------------------------------------------------------
YOUR ANSWER: A
CORRECT ANSWER: C
STATUS: ❌ Incorrect
------------------------------------------------------------
EXPLANATION:
The conflict markers work like this:

<<<<<<< HEAD
    [CODE FROM YOUR CURRENT BRANCH — where HEAD points]
=======
    [CODE FROM THE INCOMING BRANCH — being merged in]
>>>>>>> feature-branch

In this scenario:
- You're ON main (HEAD points to main)
- You're MERGING IN feature-branch

So:
- Lines between <<<<<<< HEAD and =======
  → From main (the string concatenation version)
  
- Lines between ======= and >>>>>>> feature-branch  
  → From feature-branch (the template literal version)

Why the others are wrong:
- A) Reversed — HEAD section is YOUR branch, not incoming
- B) Reversed — bottom section is INCOMING branch
- D) Never! You must remove all markers after resolving

RESOLVING CONFLICTS:
1. Decide which code to keep (or combine both)
2. Delete ALL conflict markers (<<<, ===, >>>)
3. Stage the file: git add app.js
4. Complete the merge: git commit

Resolved example (keeping template literal):
function getUserName() {
    return `${this.firstName} ${this.lastName}`;
}
