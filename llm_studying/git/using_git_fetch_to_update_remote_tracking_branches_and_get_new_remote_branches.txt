`git fetch` downloads the latest data from a remote and updates your remote-tracking branches. It does not change your local branches or working directory.

**Basic usage:**

```bash
# Fetch from the default remote (usually origin)
git fetch

# Fetch from all remotes
git fetch --all

# Fetch a specific remote
git fetch origin
```

**What it does:**

1. Contacts the remote server
2. Downloads any new commits, branches, and tags
3. Updates your remote-tracking branches (e.g., `origin/main`)
4. Leaves your local branches untouched

**How remote-tracking branches become stale:**

Your remote-tracking branches only update when you explicitly fetch. Between fetches, they're frozen in time:

```
Timeline:
---------
Monday 9am    - You run `git fetch`
Monday 10am   - Teammate pushes 3 commits to main
Monday 2pm    - Another teammate creates a new branch
Tuesday 9am   - You look at `origin/main`
              → You're seeing Monday 9am's state, missing 3 commits
              → You don't see the new branch at all
```

**Checking staleness:**

```bash
# See when you last fetched (check the modification time)
ls -la .git/FETCH_HEAD

# Compare your local view vs what's actually on remote
git remote show origin
```

**Best practices:**

```bash
# Always fetch before checking on teammates' work
git fetch --all
git log origin/<their-branch> --oneline -n 5

# Fetch and prune deleted branches
git fetch --all --prune
```

**Fetch vs. pull:**

| Command | Updates remote-tracking branches | Updates local branch |
|---------|----------------------------------|----------------------|
| `git fetch` | ✅ Yes | ❌ No |
| `git pull` | ✅ Yes | ✅ Yes (merges or rebases) |

`git fetch` is safe to run anytime — it won't affect your work in progress. It's a good habit to fetch frequently so your remote-tracking branches stay current.
