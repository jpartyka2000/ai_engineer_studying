## 📋 Question #2 — Results

**Copy the block below for your records:**

```
============================================================
GIT TUTOR — QUESTION RECORD
============================================================
Skill Level: Hard
Category: Undoing Changes & History
Type: Coding / Command
------------------------------------------------------------
QUESTION:
You accidentally committed sensitive credentials (like an 
API key) three commits ago. The commit history looks like:

a1b2c3d (HEAD -> main) Add user authentication
f4e5d6c Update README
9h8i7j6 Add API integration  <-- leaked credentials here
k1l2m3n Initial commit

You need to completely remove the commit 9h8i7j6 from 
history (not just revert it) while keeping the changes 
from the two commits that came after it.

Write the command(s) to start this process.
------------------------------------------------------------
YOUR ANSWER: git revert 9h8i7j6
CORRECT ANSWER: git rebase -i HEAD~3  (or git rebase -i k1l2m3n)
STATUS: ❌ Incorrect
------------------------------------------------------------
EXPLANATION:
git revert creates a NEW commit that undoes changes — but 
the original commit (with the credentials) remains in the 
history! Anyone with repo access could still see the leaked 
API key by browsing the commit history.

For security incidents, you must REWRITE history using:

  git rebase -i HEAD~3

This opens an interactive editor showing the last 3 commits. 
You would then change "pick" to "drop" (or delete the line) 
for commit 9h8i7j6.

After saving, Git replays the remaining commits on a new 
base, effectively erasing the bad commit from existence.

⚠️  IMPORTANT: After rewriting history, you must force push:
  git push --force-with-lease

Also consider tools like git-filter-repo for thorough 
credential scrubbing across all branches/tags.
============================================================
```

---

This is a really common interview question because it tests understanding of the difference between **revert** (safe, creates new history) vs **rebase** (dangerous, rewrites history). Security scenarios almost always require the rebase approach.
