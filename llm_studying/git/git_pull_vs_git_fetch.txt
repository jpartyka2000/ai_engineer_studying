QUESTION:
You've cloned a repository and your colleague has pushed 
new changes to the remote `main` branch. You want to see 
what changes exist on the remote WITHOUT merging them into 
your local branch yet.

Which command should you use?

A) git pull origin main
B) git fetch origin main
C) git merge origin/main
D) git remote update --pull
------------------------------------------------------------
YOUR ANSWER: A
CORRECT ANSWER: B
STATUS: ❌ Incorrect
------------------------------------------------------------
EXPLANATION:
The key distinction is fetch vs pull:

- git fetch: Downloads commits, files, and refs from the 
  remote but does NOT modify your working directory or 
  current branch. It's a "safe" way to see what's new.

- git pull: Essentially runs "git fetch" + "git merge" in 
  one step. It WILL merge remote changes into your local 
  branch automatically.

Think of it this way:
  git pull = git fetch + git merge

After fetching, you can safely inspect changes with:
  git log HEAD..origin/main     (see commit list)
  git diff HEAD origin/main     (see actual changes)

Why the others are wrong:
- A) Pulls AND merges — not what we want
- C) Only merges (doesn't fetch new data from remote)
- D) Invalid command syntax
