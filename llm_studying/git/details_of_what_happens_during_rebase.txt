**How rebasing rewrites your commits**

When you rebase, Git doesn't move your existing commits—it creates entirely new commits.

**What happens during a rebase:**

1. Git identifies your branch's commits that aren't in the target branch (e.g., `origin/main`)
2. Git temporarily saves the *changes* (diffs) from each of those commits
3. Git resets your branch to point to the target branch
4. Git replays each saved change as a *new* commit on top

**Why the commits are "new":**

A commit's hash is determined by its content *and* its parent commit. When you rebase onto `origin/main`, your commits now have different parents than before—so they get entirely new hashes, even if the code changes are identical.

Before rebase:
```
main:        A -- B -- C
your-branch:      \-- D -- E -- F
```

After rebase:
```
main:        A -- B -- C
your-branch:              \-- D' -- E' -- F'
```

`D'`, `E'`, and `F'` contain the same changes as `D`, `E`, and `F`, but they're different commits with different hashes because their parent is now `C` instead of `B`.

**This is why force push is required:**

The remote still has `D -- E -- F`. Your local branch now has `D' -- E' -- F'`. Git sees these as completely different commits, so you must force push to replace the old history with the new one.
