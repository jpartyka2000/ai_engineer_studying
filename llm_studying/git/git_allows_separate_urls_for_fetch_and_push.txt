Git allows you to configure different URLs for fetching and pushing to the same remote. That's why they're shown separately.

**Typical output (same URL for both):**

```
origin  git@github.com:yourname/repo.git (fetch)
origin  git@github.com:yourname/repo.git (push)
```

Most of the time they're identical, but they don't have to be.

**When different URLs are useful:**

1. **Fork workflow:**
   ```
   origin    git@github.com:yourname/fork.git (fetch)
   origin    git@github.com:yourname/fork.git (push)
   upstream  git@github.com:original/repo.git (fetch)
   upstream  git@github.com:original/repo.git (push)
   ```
   You fetch from the original repo to stay updated, but push to your fork.

2. **Read-only access:**
   ```
   origin  https://github.com/company/repo.git (fetch)
   origin  git@github.com:company/repo.git (push)
   ```
   Fetch over HTTPS (no auth needed for public repos), push over SSH (requires auth).

3. **Mirror or caching:**
   ```
   origin  https://internal-mirror.company.com/repo.git (fetch)
   origin  git@github.com:company/repo.git (push)
   ```
   Fetch from a fast internal mirror, push to the actual remote.

**Setting different URLs:**

```bash
# Set fetch URL
git remote set-url origin <fetch-url>

# Set push URL separately
git remote set-url --push origin <push-url>
```

In practice, most developers never change these and both URLs stay the same. But the flexibility exists for more complex workflows.
