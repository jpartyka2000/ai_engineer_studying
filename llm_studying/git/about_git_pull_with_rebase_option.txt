`git pull --rebase` fetches remote changes and replays your local commits on top of them, instead of creating a merge commit.

**Visual difference:**

```
Regular git pull (merge):

      C---D  (your local commits)
     /     \
A---B-------E  (merge commit)
     \
      F---G  (remote commits)

git pull --rebase:

A---B---F---G---C'---D'  (your commits replayed on top)
```

**Why use it:**
- Keeps history linear and clean (no merge commits for every pull)
- Easier to read `git log`
- Your commits appear after the remote changes, as if you started from the latest code

**When NOT to use it:**
- If you've already pushed your commits and others are working off them (rebase rewrites commit hashes)
- If you have complex merge conflicts that would be easier to resolve once in a merge

**Make it the default** (if you like it):
```bash
git config --global pull.rebase true
```

**If conflicts occur during rebase:**
```bash
# Fix the conflicts, then:
git add <fixed-files>
git rebase --continue

# Or abort and go back to before the pull:
git rebase --abort
```
