A tree in Git is an object that represents a directory. It maps filenames to blobs (file contents) or other trees (subdirectories).

**What a tree contains:** A list of entries, where each entry has a mode (file permissions), a type (blob or tree), a SHA-1 hash pointing to the object, and a name.

**Example structure:**
```
tree 8a3f2b...
├── README.md      -> blob 4c1a9e...
├── main.py        -> blob 7d2f8c...
└── src/           -> tree 2b9e4a...
    ├── utils.py   -> blob 1f3d7b...
    └── config.py  -> blob 9e2c5a...
```

**Why trees matter:** Blobs have no concept of filenames or location — they're just content. Trees provide the structure. When you commit, Git creates a tree that captures the entire directory state at that moment. The commit object then points to that root tree.

**How Git builds them:** When you stage files (`git add`), Git creates blobs for the content and updates an index. At commit time, Git constructs tree objects from that index, nesting them to match your directory hierarchy.

**Efficiency:** If a file hasn't changed between commits, the new tree just points to the same blob. If a whole subdirectory is unchanged, the new root tree points to the same subtree. This makes Git's storage very efficient — identical content is never duplicated.
