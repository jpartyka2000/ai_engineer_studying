Great question! Here's how the different `git reset` modes work:

**`git reset --soft <commit>`**

Moves the HEAD pointer to the specified commit, but leaves both the staging area (index) and working directory unchanged. All the changes from the "undone" commits appear as staged, ready to be committed again. Useful when you want to squash commits or redo a commit message.

**`git reset --mixed <commit>`** (this is the default)

Moves HEAD to the specified commit and resets the staging area to match, but leaves your working directory unchanged. Changes from undone commits appear as unstaged modifications. This is what happens when you run plain `git reset` without a flag.

**`git reset --hard <commit>`**

The most destructive option. Moves HEAD, resets the staging area, *and* overwrites your working directory to match the specified commit. All uncommitted changes (staged or unstaged) are permanently lost. Use with caution.

**Quick comparison:**

| Mode | HEAD | Staging Area | Working Directory |
|------|------|--------------|-------------------|
| `--soft` | ✓ moved | unchanged | unchanged |
| `--mixed` (default) | ✓ moved | ✓ reset | unchanged |
| `--hard` | ✓ moved | ✓ reset | ✓ reset |

**Practical examples:**

```bash
# Undo last commit but keep changes staged
git reset --soft HEAD~1

# Unstage files but keep changes in working directory
git reset HEAD~1        # same as --mixed

# Completely discard last commit and all changes
git reset --hard HEAD~1
```

The key takeaway: `--soft` preserves the most, `--hard` preserves the least, and `--mixed` (the default) is the middle ground.
