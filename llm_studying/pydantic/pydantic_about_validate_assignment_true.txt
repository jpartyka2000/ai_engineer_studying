# The `validate_assignment=True` Configuration

## What It Does:

`validate_assignment=True` makes Pydantic run validation whenever you assign a new value to a field **after** the model is created. Without it, post-creation assignments bypass all validation!

## Basic Example - The Difference:

```python
from pydantic import BaseModel, ConfigDict, field_validator

# WITHOUT validate_assignment (default)
class UnsafeModel(BaseModel):
    age: int
    
    @field_validator("age")
    @classmethod
    def check_age(cls, v):
        if v < 0 or v > 150:
            raise ValueError("Invalid age")
        return v

unsafe = UnsafeModel(age=25)  # Validator runs ✓
unsafe.age = -999  # No validation! This works! 😱
unsafe.age = "not a number"  # No validation! This works too! 😱
print(unsafe.age)  # "not a number" - Model is corrupted!

# WITH validate_assignment
class SafeModel(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    age: int
    
    @field_validator("age")
    @classmethod
    def check_age(cls, v):
        if v < 0 or v > 150:
            raise ValueError("Invalid age")
        return v

safe = SafeModel(age=25)  # Validator runs ✓
safe.age = -999  # ValidationError! ✓
safe.age = "not a number"  # ValidationError! ✓
```

## Type Coercion Still Works:

```python
class Model(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    count: int
    tags: List[str]

m = Model(count=1, tags=["a"])

# These work - Pydantic converts types
m.count = "42"  # String → int conversion happens
print(m.count)  # 42 (int)

m.tags = ("b", "c")  # Tuple → List conversion
print(m.tags)  # ["b", "c"]

# This fails - can't convert
m.count = "not a number"  # ValidationError!
```

## Performance Implications:

```python
class ExpensiveModel(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    data: Dict[str, Any]
    
    @field_validator("data")
    @classmethod
    def expensive_check(cls, v):
        # Imagine this hits a database or API
        print("Running expensive validation...")
        time.sleep(0.1)
        return v

model = ExpensiveModel(data={})  # Validation runs once

# Without validate_assignment: instant
# With validate_assignment: runs expensive check every time!
for i in range(100):
    model.data = {"iteration": i}  # 100 validations = 10 seconds!
```

## Common Use Cases:

### 1. **Maintaining Data Integrity in Long-Lived Objects**
```python
class ConfigManager(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    max_connections: int
    timeout: float
    
    @field_validator("max_connections")
    @classmethod
    def validate_connections(cls, v):
        if v < 1 or v > 1000:
            raise ValueError("Connections must be 1-1000")
        return v

config = ConfigManager(max_connections=10, timeout=30.0)
# Later in your application...
config.max_connections = 5000  # Prevented! Keeps config valid
```

### 2. **Stateful Models That Change Over Time**
```python
class GameState(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    health: int
    position: Tuple[int, int]
    
    @field_validator("health")
    @classmethod
    def validate_health(cls, v):
        return max(0, min(100, v))  # Clamp between 0-100

player = GameState(health=100, position=(0, 0))
player.health = 150  # Automatically clamped to 100
player.health = -10  # Automatically clamped to 0
```

### 3. **API Request/Response Models**
```python
class APIRequest(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    endpoint: str
    method: Literal["GET", "POST", "PUT", "DELETE"]
    headers: Dict[str, str] = {}
    
    @field_validator("endpoint")
    @classmethod
    def validate_endpoint(cls, v):
        if not v.startswith("/"):
            v = "/" + v
        return v

req = APIRequest(endpoint="users", method="GET")
req.endpoint = "posts"  # Automatically prepends "/"
req.method = "PATCH"  # ValidationError! Not in Literal
```

## Gotchas and Edge Cases:

### 1. **Bypassing Validation Still Possible**
```python
class Model(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    value: int

m = Model(value=1)
m.value = 2  # Validated ✓

# These bypass validation even with validate_assignment!
m.__dict__["value"] = "bypass!"  # Direct dict access
object.__setattr__(m, "value", "also bypass!")  # Direct setattr
```

### 2. **Interdependent Fields**
```python
class Rectangle(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    width: float
    height: float
    area: float = 0
    
    @field_validator("area")
    @classmethod
    def calculate_area(cls, v, info):
        # Problem: When setting width/height, area doesn't auto-update!
        # This validator only runs when area itself is set
        return v

rect = Rectangle(width=10, height=5, area=50)
rect.width = 20  # area is still 50, not 100!
```

### 3. **Private Attributes Not Validated**
```python
class Model(BaseModel):
    model_config = ConfigDict(validate_assignment=True)
    
    public: int
    _private: int = PrivateAttr(default=0)

m = Model(public=1)
m.public = "2"  # Validated, converted to int
m._private = "not validated"  # No validation on private attrs!
```

## When to Use It:

**✅ Use `validate_assignment=True` when:**
- Models are modified after creation
- You need runtime type safety
- Working with user input that changes over time
- Building stateful applications
- Config or settings objects

**❌ Avoid it when:**
- Models are immutable after creation
- Performance is critical (validation overhead)
- You're only using models for serialization/deserialization
- Working with trusted, pre-validated data

## Best Practice Pattern:

```python
class MutableModel(BaseModel):
    """For models that change after creation"""
    model_config = ConfigDict(validate_assignment=True)
    # ... fields ...

class ImmutableModel(BaseModel):
    """For models that are create-once"""
    model_config = ConfigDict(frozen=True)  # Alternative: make immutable!
    # ... fields ...

class HighPerfModel(BaseModel):
    """For performance-critical paths with trusted data"""
    # No validate_assignment, minimal validators
    # ... fields ...
```

The key insight: `validate_assignment=True` trades performance for safety. It's excellent for maintaining data integrity in mutable models but adds overhead to every assignment!
