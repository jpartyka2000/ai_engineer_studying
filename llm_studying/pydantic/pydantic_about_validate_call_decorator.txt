### ✅ Using `@validate_call` in Pydantic

In addition to data models, Pydantic can also validate **function arguments** using the `@validate_call` decorator. This makes it easy to ensure that functions always receive properly typed inputs.

---

### 🔎 What It Does

* Wraps a function in a Pydantic-powered validator.
* At runtime, whenever the function is called:

  1. The arguments are validated against the function’s type hints.
  2. If valid, the function executes normally (with values coerced if needed).
  3. If invalid, a `ValidationError` is raised.

---

### 📌 Example

```python
from pydantic import validate_call

@validate_call
def greet(name: str, times: int = 1) -> str:
    return " ".join([f"Hello, {name}!" for _ in range(times)])

print(greet("Alice", 2))      # ✅ "Hello, Alice! Hello, Alice!"
print(greet(name="Bob"))      # ✅ "Hello, Bob!"
print(greet(123))             # ❌ ValidationError (name must be str)
```

Here:

* `"Alice"` and `2` are validated and passed in.
* If you pass the wrong type (like `123` for `name`), Pydantic raises an error before the function body runs.

---

### ⚖️ Features

* **Type coercion**: Strings can be cast to ints, floats, etc., if possible.
* **Default handling**: Works with default values just like normal Python functions.
* **Complex types**: Supports `Optional`, `List`, `Dict`, `Literal`, `TypedDict`, `BaseModel`, and more.
* **Performance**: Slight overhead (since it builds a `FunctionModel` under the hood), but ensures strict correctness.

---

### 📊 Use Cases

* API endpoint functions (validate request parameters).
* Data processing pipelines (validate inputs at each step).
* Utility functions where correctness matters more than raw speed.

---

### ✅ Takeaway

The `@validate_call` decorator lets you use Pydantic’s validation on **function arguments**, giving you:

* Safer, self-documenting APIs.
* Automatic type coercion.
* Early, clear error messages.

It’s a great way to extend Pydantic beyond just models and enforce correctness throughout your code.
