A **flamegraph** is a visualization that shows you *where a program spends its time*—typically in CPU profiling, but also for I/O, memory, or other resources. It was popularized by Brendan Gregg as a way to make sense of large stacks of profiling data.

---

## 🔥 How to Read a Flamegraph

* **Stacks are shown vertically**

  * The bottom row = functions at the root (e.g., `main`).
  * Each row above = functions that were called by the row below.
  * The taller the “flame,” the deeper the call stack.

* **Width encodes time**

  * The wider a box, the *more total time* (across samples) was spent in that function or its callees.
  * So the *widest areas* are your hot spots.

* **Colors**

  * Colors don’t have meaning by default (they’re usually chosen to look like flames).
  * Some tools use different colors for system vs. user code, but usually, color is just decoration.

* **Direction**

  * X-axis = *not ordered by time*. Boxes are laid out alphabetically or arbitrarily to fit.
  * Y-axis = stack depth, not time. The graph doesn’t show “when” things happened, but “where” CPU time accumulated.

---

## 🛠 Example

Imagine you profile a Python script with `py-spy`:

```bash
py-spy record -o profile.svg -- python myscript.py
```

Opening `profile.svg` in a browser, you might see:

```
main
 └── load_data
       └── parse_file
             └── regex_match
```

If `regex_match` takes most of the time, its box will be wide. That tells you optimizing the regex will yield the biggest performance gain.

---

## ✅ Why Flamegraphs Are Useful

* They **compress millions of samples** into a single intuitive picture.
* You can quickly see **which code paths dominate**.
* They work across languages (Python, C, Java, Go, Rust).
* Great for **performance tuning** in both development and production.

---

👉 In short:
A **flamegraph is a heatmap of your stack traces**, showing visually where your program spends time. The “tall flames” show stack depth, and the “wide flames” show where the CPU is burning the most cycles.
