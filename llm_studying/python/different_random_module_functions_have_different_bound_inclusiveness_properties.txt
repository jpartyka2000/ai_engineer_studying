No, the bounds are **NOT** always inclusive in Python's `random` module. Different functions have different boundary behaviors:

## Inclusive vs Exclusive Bounds:

### **INCLUSIVE on both ends:**
- `random.randint(a, b)` - Returns integer from [a, b] inclusive
  - Example: `random.randint(1, 10)` can return 1, 2, ..., 10

### **INCLUSIVE start, EXCLUSIVE end:**
- `random.randrange(start, stop)` - Returns integer from [start, stop)
  - Example: `random.randrange(1, 10)` can return 1, 2, ..., 9 (NOT 10)
- `random.choice(seq)` - Uses standard Python indexing [0, len(seq))

### **EXCLUSIVE on both ends (for continuous):**
- `random.random()` - Returns float in (0.0, 1.0)
  - Technically can return 0.0 but never exactly 1.0
- `random.uniform(a, b)` - Returns float in [a, b] or [b, a]
  - Actually inclusive on both ends due to floating-point rounding

## Quick Reference:
```python
random.randint(1, 10)      # 1, 2, 3, ..., 10 (BOTH inclusive)
random.randrange(1, 10)    # 1, 2, 3, ..., 9  (end EXCLUSIVE)
random.random()            # 0.0 <= x < 1.0   (end EXCLUSIVE)
random.uniform(1, 10)      # 1.0 <= x <= 10.0 (BOTH inclusive*)
```

## Rule of Thumb:
- `randint` = "inclusive integer"
- `randrange` = works like Python's `range()` (exclusive end)
- Floating-point functions are generally exclusive of 1.0

The inconsistency is a common source of bugs, so always double-check the documentation for the specific function you're using!
