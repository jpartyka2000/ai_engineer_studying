## Key Differences

| Aspect | `@classmethod` | `@staticmethod` |
|--------|----------------|-----------------|
| First argument | `cls` (the class) | Nothing passed automatically |
| Access to class | Yes | No |
| Access to instance | No | No |
| Can modify class state | Yes | No |
| Inheritance behavior | Gets subclass when called on subclass | No awareness of class hierarchy |

## Example

```python
class Parent:
    name = "Parent"
    
    @classmethod
    def class_method(cls):
        return f"Called on {cls.name}"
    
    @staticmethod
    def static_method():
        return "No access to class or instance"

class Child(Parent):
    name = "Child"

# Class method knows which class called it
Parent.class_method()  # "Called on Parent"
Child.class_method()   # "Called on Child"

# Static method is just a namespaced function
Parent.static_method() # "No access to class or instance"
Child.static_method()  # "No access to class or instance"
```

## Common use cases

**`@classmethod`** — Factory methods / alternate constructors:
```python
class Date:
    def __init__(self, year, month, day):
        self.year, self.month, self.day = year, month, day
    
    @classmethod
    def from_string(cls, date_str):
        year, month, day = map(int, date_str.split('-'))
        return cls(year, month, day)  # Works correctly with subclasses
```

**`@staticmethod`** — Utility functions that logically belong to the class but don't need class/instance access:
```python
class Math:
    @staticmethod
    def add(a, b):
        return a + b
```

## When to use which

- Need to access or modify class state? → `@classmethod`
- Creating alternate constructors? → `@classmethod`
- Just a utility function grouped with the class? → `@staticmethod`
- Could it be a standalone function? → `@staticmethod` (or just a regular function)
