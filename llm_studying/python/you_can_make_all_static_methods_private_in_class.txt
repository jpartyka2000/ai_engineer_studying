**Considering Python method naming conventions**

The user is asking about the use of underscores to indicate private methods for a class composed entirely of static methods.

In Python, a leading underscore is a convention for internal or private methods. It’s not enforced by the interpreter, but it signals that the method is intended for internal use. A double underscore triggers name mangling. While you don't "need" to use underscores, it’s a good practice if you want to discourage external usage. Static methods can also follow this convention.


Thought for 8 seconds


Python won’t enforce “privacy” on any methods—even static ones—so the leading underscore is purely a **convention** to signal “internal use only.” You **don’t have to** name your helpers with an underscore, but it’s strongly recommended if you want readers (and linters, IDEs, etc.) to treat them as private.

```python
class TransformCustomerData:
    @staticmethod
    def transform_data(df):
        df = TransformCustomerData._clean(df)   # calls a “private” helper
        df = TransformCustomerData._normalize(df)
        return df

    @staticmethod
    def _clean(df):
        # internal step  
        ...

    @staticmethod
    def _normalize(df):
        # internal step  
        ...
```

* **Single underscore** (`_method`) is the conventional way to mark “don’t call this from outside.”
* **Double underscore** (`__method`) triggers name‑mangling (`_ClassName__method`), but is rarely worth the extra complexity here.

If all your logic really is private except the one entry‐point, using the underscore makes your API clearer. Otherwise, you can omit it—you’ll still get the same behavior.
