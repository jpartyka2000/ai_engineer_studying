Here’s a dedicated post on **`NotRequired`**:

---

### 🔑 What is `NotRequired` in Python Typing?

`NotRequired` is a keyword from **PEP 655** that you can use with `TypedDict` to mark **individual keys as optional**.

It lives in the **`typing_extensions`** module (and in the built-in `typing` module starting with Python 3.11).

---

### 📌 Why It Exists

Before PEP 655, you only had two options with `TypedDict`:

1. All keys required (`total=True`, the default).
2. All keys optional (`total=False`).

But sometimes you want **a mix**:

* Some keys **must always be present**.
* Others are **optional**.

That’s where `NotRequired` (and its opposite, `Required`) comes in.

---

### 🧩 Example

```python
from typing_extensions import TypedDict, NotRequired

class User(TypedDict):
    id: int                # required
    name: str              # required
    nickname: NotRequired[str]   # optional
```

Usage:

```python
user1: User = {"id": 1, "name": "Alice"}               # ✅ valid
user2: User = {"id": 2, "name": "Bob", "nickname": "B"}  # ✅ valid
user3: User = {"id": 3}                                # ❌ missing required 'name'
```

---

### ⚖️ `NotRequired` vs `total=False`

* `NotRequired` → fine-grained control per key.
* `total=False` → makes **all keys** optional.

```python
class PartialUser(TypedDict, total=False):
    id: int
    name: str
```

Here, both `id` and `name` are optional.

---

### ✅ Takeaway

* `NotRequired` lets you declare optional fields in a `TypedDict` **without making the entire dict optional**.
* Use it when modeling real-world data (like API responses) where some fields are always present, but others may or may not appear.
