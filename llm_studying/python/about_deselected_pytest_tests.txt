In **pytest**, a test being **deselected** means that it was **intentionally skipped from execution** based on filtering criteria. This usually happens when you use the `-k` or `-m` options to select specific tests or when `pytest` dynamically filters out certain tests.  

### **Common Reasons for Test Deselection**
1. **Using `-k` to select specific test names**  
   - If you run `pytest -k "some_keyword"`, only tests matching `"some_keyword"` will run, and others will be **deselected**.
   - Example:
     ```sh
     pytest -k "login"
     ```
     - This runs only tests with `"login"` in their name.
     - All other tests are **deselected**.

2. **Using `-m` to run tests with specific markers**  
   - If you mark tests with `@pytest.mark.slow` and run only `"slow"` tests, others get **deselected**.
   - Example:
     ```sh
     pytest -m slow
     ```
     - Only tests marked `@pytest.mark.slow` will run.
     - Other tests will be **deselected**.

3. **Explicitly skipping tests in a `pytest.ini`, `conftest.py`, or CLI options**  
   - Example (`pytest.ini`):
     ```ini
     [pytest]
     addopts = -m "not slow"
     ```
     - This automatically deselects `@pytest.mark.slow` tests.

4. **Test collection filtering**  
   - If pytest fails to recognize a test file/function due to:
     - A mismatch in naming conventions (`test_*.py` or `*_test.py`).
     - A missing `pytest` discovery pattern.
     - A dynamic test discovery rule in a plugin.

### **How to Check Deselected Tests**
If tests are deselected, `pytest` will show output like:
```
collected 10 items / 5 deselected / 5 selected
```
To explicitly see **deselected** tests:
```sh
pytest --deselect
```
