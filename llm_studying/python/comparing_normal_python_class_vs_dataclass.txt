Yes! You **can absolutely** define getters and setters using `@property` in a **normal Python class** just like in a `dataclass`. The main advantage of `dataclass` is reducing boilerplate when working with simple data structures.  

### **Comparing a Normal Class vs. a `dataclass`**
#### **1️⃣ Normal Class with Getters & Setters**
```python
class Person:
    def __init__(self, name: str, age: int):
        self.name = name
        self._age = age  # Conventionally private

    @property
    def age(self):  # Getter
        return self._age

    @age.setter
    def age(self, value):  # Setter with validation
        if value < 0:
            raise ValueError("Age cannot be negative")
        self._age = value

p = Person("Alice", 30)
print(p.age)    # 30
p.age = 31      # Works fine
print(p.age)    # 31
```
✔ **Explicit control over methods and attributes.**  
❌ **More boilerplate** (manual `__init__`, explicit attributes, and method definitions).

---

#### **2️⃣ Using `dataclass` with Getters & Setters**
```python
from dataclasses import dataclass, field

@dataclass
class Person:
    name: str
    _age: int = field(repr=False)  # Conventionally private

    @property
    def age(self):  # Getter
        return self._age

    @age.setter
    def age(self, value):  # Setter with validation
        if value < 0:
            raise ValueError("Age cannot be negative")
        self._age = value

p = Person("Alice", 30)
print(p.age)    # 30
p.age = 31      # Works fine
```
✔ **Less boilerplate** (no need for explicit `__init__`).  
✔ **Built-in `__repr__`, `__eq__`, etc., for free**.  
✔ **Same control over property logic as a normal class**.  

---

### **So, When is `dataclass` Better?**
1. **When you want to reduce boilerplate**  
   - If you're mainly storing data (like models, configs, DTOs), `dataclass` is cleaner.  
2. **When you want auto-generated methods (`__init__`, `__repr__`, `__eq__`)**  
   - Saves time compared to manually defining these in a normal class.  
3. **When immutability is needed (`frozen=True`)**  
   - Normal classes require extra work to enforce immutability.  

But if your class has **complex behavior, lots of methods, or deep inheritance**, a normal class might be better.
