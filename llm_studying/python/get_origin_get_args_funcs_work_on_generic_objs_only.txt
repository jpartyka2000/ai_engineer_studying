No—both `get_origin()` and `get_args()` only work on *type objects* (the things you write in annotations), not on actual instances at runtime. If you do this:

```python
from typing import get_origin, get_args

data = [1, 2, 3]
print(get_origin(data))     # → None
print(get_args(data))       # → ()
```

you’ll just get `(None, ())`, because `data` isn’t a `typing`-generic, it’s a *list instance*.

---

## What you should pass in

1. **Subscripted generics**, e.g.:

   ```python
   from typing import get_origin, get_args

   # Works:
   T = list[int]
   print(get_origin(T))       # → <class 'list'>
   print(get_args(T))         # → (<class 'int'>,)
   ```

2. **Annotations you’ve pulled via** `get_type_hints()`:

   ```python
   from typing import get_type_hints, get_origin, get_args

   def fn(x: dict[str, float]): ...
   hints = get_type_hints(fn)
   ann = hints['x']           # dict[str, float]
   print(get_origin(ann))     # → <class 'dict'>
   print(get_args(ann))       # → (<class 'str'>, <class 'float'>)
   ```

---

## Runtime inspection of instances

If you really need to inspect an instance’s “inner” types at runtime, you’ll have to fall back to the pattern you saw before:

```python
data = [1, 2, 3]
outer_type = type(data)                # <class 'list'>
inner_types = { type(item) for item in data }  
# → {<class 'int'>}

# For a dict:
d = {"a":1, "b":2}
outer_type  = type(d)                  # <class 'dict'>
key_types   = { type(k) for k in d }   # {<class 'str'>}
value_types = { type(v) for v in d.values() }  # {<class 'int'>}
```

---

### Why generics vs. instances?

* **Generics** (`list[int]`, `dict[str, bytes]`, etc.) carry their “type arguments” in metadata that `get_origin`/`get_args` can peel back.
* **Plain instances** (actual lists, dicts, tuples…) have *no* memory of any generic parameters at runtime—Python erases that after you construct them.

So whenever you need `get_origin`/`get_args`, make sure you’re passing it a *type*, not a *value*.
