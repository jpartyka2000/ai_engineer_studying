Great question 👌 — these two often get confused, but they serve **different purposes** in the testing pyramid.

---

## 🔎 Unit Test

* **Scope:** Tests the **smallest piece of code** in isolation (usually a single function or class method).
* **Goal:** Verify that a unit of code behaves correctly given specific inputs.
* **Dependencies:**

  * Should avoid external dependencies (databases, APIs, file systems).
  * If needed, dependencies are replaced with **mocks** or **stubs**.
* **Speed:** Very fast — meant to be run often (like on every code change).

✅ Example (Python):

```python
def add(a, b):
    return a + b

def test_add():
    assert add(2, 3) == 5
```

This test checks **only** the `add` function logic.

---

## 🔎 Integration Test

* **Scope:** Tests how **multiple components work together**.
* **Goal:** Verify that interactions between modules, or between your code and external systems, behave as expected.
* **Dependencies:**

  * Usually includes “real” dependencies (databases, APIs, message queues, etc.).
  * Sometimes runs in a controlled environment (like a test database).
* **Speed:** Slower than unit tests (more setup/teardown).

✅ Example:

```python
def test_user_created_in_db(test_db):
    user_id = create_user(test_db, "alice")
    user = get_user(test_db, user_id)
    assert user.name == "alice"
```

This test checks the integration between the **user creation code** and the **database layer**.

---

## ⚖️ Key Differences

| Aspect           | Unit Test                          | Integration Test                       |
| ---------------- | ---------------------------------- | -------------------------------------- |
| **Focus**        | Single function/class              | Interaction of multiple components     |
| **Dependencies** | Mocked or isolated                 | Real or realistic (DB, APIs, services) |
| **Speed**        | Very fast                          | Slower (setup/teardown required)       |
| **Failures**     | Pinpoints bug in specific function | Shows system-level issues (boundaries) |
| **Frequency**    | Run constantly (every commit)      | Run less frequently (nightly, CI/CD)   |

---

## ✅ Takeaway

* **Unit tests** → verify that each building block works correctly on its own.
* **Integration tests** → verify that the blocks work correctly together.

Both are essential: unit tests catch bugs early and cheaply, while integration tests ensure your system actually works in real-world conditions.
