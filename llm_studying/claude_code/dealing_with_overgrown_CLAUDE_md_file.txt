This is a common challenge as projects grow! Here are some practical strategies:

**Split into multiple files**

Instead of one large CLAUDE.md, create a structured documentation system:

```
/docs
  CLAUDE.md          # High-level overview, core principles, quick reference
  ARCHITECTURE.md    # System design, component relationships
  CONVENTIONS.md     # Coding standards, naming conventions
  API.md             # API contracts, data structures
  CHANGELOG.md       # Recent changes (can be pruned periodically)
```

Then reference these in your main CLAUDE.md: "For architecture details, see docs/ARCHITECTURE.md"

**Prioritize ruthlessly in CLAUDE.md**

Keep only what Claude Code needs *most frequently*:
- Current project state and immediate goals
- Critical conventions that are easy to forget
- Known gotchas and past mistakes to avoid
- Key file locations

Move historical context and stable documentation elsewhere.

**Use a "working memory" section**

Add a section at the top for current sprint/task context that you update frequently, while keeping stable reference material below or in separate files.

**Prune regularly**

- Remove completed tasks and resolved issues
- Consolidate repetitive entries
- Archive old decisions that are now "baked in" to the codebase

**Let the code be self-documenting**

If something is well-established in the codebase, you may not need it in CLAUDE.md anymore. Claude Code can read the actual code.
