Under the hood LightGBM can build its histograms in two different ways—**column‑wise** and **row‑wise**—and by default it will benchmark a small chunk of your data and pick whichever is faster. Unfortunately:

1. **Auto‑selection can flip between runs.**
   The tiny “speed test” at startup sometimes chooses column‑wise and other times row‑wise (depending on thread timing, CPU load, etc.), so you end up with a different algorithm each run. ([GitHub][1])

2. **Column‑wise is non‑deterministic when multi‑threaded.**
   In column‑wise mode the histograms for each bin are summed across many threads in parallel.  Floating‑point reductions in different thread orders give slightly different bin totals, which—even if infinitesimal—can change which split has the highest gain. ([GitHub][1])

3. **Row‑wise is fully sequential (per row) and bit‑for‑bit repeatable.**
   When you force row‑wise, LightGBM does one pass over your rows in a fixed order, accumulating gradients/hessians in a single thread‑coordinated loop.  That removes both the “which algorithm?” nondeterminism and the “which thread ordered the sums?” nondeterminism.

4. **Interaction with `deterministic=true`.**
   If you ask for full reproducibility (`deterministic=true`), the docs explicitly tell you to also set **either** `force_row_wise=true` **or** `force_col_wise=true`, otherwise you’ll still get those tiny numerical instabilities from auto‑choice. ([LightGBM Documentation][2])

---

### So, to guarantee bit‑wise identical models and predictions:

```python
model = LGBMClassifier(
    ...,
    deterministic=True,
    force_row_wise=True,   # ⇐ fixes histogram builder to the reproducible, single‑pass algorithm
    random_state=42,
    n_jobs = <any>
)
```

With that in place, **every run** on the **same data**, with the **same seed**, **same code**, and **same number of threads**, will build **exactly** the same trees and give **identical** predictions.
