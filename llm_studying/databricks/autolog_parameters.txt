Yes, many of MLflow’s autologging integrations offer parameters that let you control which artifacts are automatically logged. For example, when using MLflow’s autologging for scikit-learn via `mlflow.sklearn.autolog()`, you can pass parameters such as:

- **`log_input_examples`**: If set to `True`, MLflow logs a sample of the input data as an artifact. This can help you later understand what kind of data the model was trained on.
- **`log_model_signature`**: When enabled, MLflow attempts to infer and log a model signature (i.e. the input and output schema), which is useful for model serving and reproducibility.
- **`log_models`**: Determines whether the model itself is logged as an artifact.
  
Other frameworks (like XGBoost, LightGBM, PyTorch, etc.) have similar autolog parameters that allow you to control artifacts such as evaluation plots, feature importance, or even intermediate outputs. 

For instance, with `mlflow.xgboost.autolog()`, you might control parameters related to how the model is logged, and in some cases, autolog can capture additional visualizations like feature importance plots.

If you want to log additional artifacts beyond what autolog captures automatically—such as custom plots or data files—you can always call `mlflow.log_artifact()` or `mlflow.log_figure()` manually after autologging has done its work.

In summary, autologging methods do provide parameters to fine-tune which artifacts get logged automatically, though the exact parameters depend on the ML framework you’re using.
