Yes—injecting secrets via the Databricks Secrets API into environment variables is generally quite secure, because:

1. **Encrypted at rest**

   * Secrets you store in a scope are kept encrypted in Databricks’ backing store (e.g. Azure Key Vault, AWS KMS, or the Databricks-managed secret store).

2. **Masked in the UI and logs**

   * When you configure the cluster’s env-vars, you only ever see `{{secrets/aws-creds/aws_access_key_id}}`—the actual key is never shown.
   * At runtime, Databricks masks any printed value (`print(os.environ["AWS_ACCESS_KEY_ID"])` will show `****…`) and will scrub it from standard driver logs.

3. **Granular access control**

   * Only users with explicit **READ** permission on the secret scope can retrieve the values via `dbutils.secrets.get`.
   * Users without that permission (even if they can edit cluster config) cannot see or use the secret.

4. **No plain-text storage**

   * The keys are never committed to notebook source or persisted in DBFS/Git—they live only in the secret scope and in-memory as env-vars on the driver/executors.

---

### Caveats & best practices

* **Don’t print or log env-vars**
  Accidentally doing `print(os.environ["AWS_SECRET_ACCESS_KEY"])` in your code will expose it in driver stdout (though still masked by default).

* **Limit cluster-config permissions**
  Use Databricks Cluster Policies to restrict who can modify env-vars on production clusters.

* **Rotate regularly**
  Even with strong controls, periodically rotate secrets in your scope and update your clusters.

* **Prefer instance profiles**
  If you can, attach an AWS IAM role instead of managing keys—then there’s no secret in your scope at all.

But under the hood, Databricks’ secret-scope mechanism, masking, and permission model ensure that your AWS keys never appear in clear text in the UI or logs, making this a secure way to supply them to your jobs.
