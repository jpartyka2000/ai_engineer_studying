Yes—any metrics you **report** as evidence of your model’s real‐world performance should come from data the model hasn’t “seen” during training or tuning. In practice that means:

1. **Validation set metrics**  
   - When you’re iterating and tuning hyperparameters, you measure F1, accuracy, etc., on a **validation set** (or via cross‐validation) to decide which model/config is best.  
   - Those validation metrics drive your model development, but they’re still “out‐of‐sample” relative to the training data.

2. **Test set metrics**  
   - Once you’ve locked in your model and hyperparameters, you evaluate on a separate **test set** (never used anywhere else) to get your final F1, accuracy, ROC AUC, etc.  
   - Those test metrics are what you report in papers, dashboards, or stakeholder presentations as the model’s true expected performance.

3. **Cross‐validation summaries**  
   - If you use K‐fold CV without a held‐out test set, you typically report the **mean and standard deviation** of your metrics across folds (e.g. “mean F1 = 0.82 ± 0.03”).  
   - These CV‐averaged scores are also out‐of‐fold, and can serve as your “validation” performance.

---

### Why not training‐set metrics?

- **Training metrics** overstate performance—you’ve optimized the model to fit those examples, so you’ll always look better there.  
- **Validation/test metrics** approximate how your model will fare on truly unseen data.

---

#### Quick guideline

| Phase             | Data used                    | Metrics purpose                   |
|-------------------|------------------------------|-----------------------------------|
| **Training**      | Training set only            | Parameter fitting                 |
| **Validation**    | Validation set (or CV)       | Hyperparameter tuning, model selection |
| **Testing / Report** | Test set (held out)      | Final performance reporting       |

So when you say “our model achieved F1 = 0.78,” it should always refer to your validation or test set—not the training set.
