MLflow doesn’t automatically produce separate versions of the same model for each library. Instead, it logs the model along with metadata that records the "flavor" or framework it was built with (such as sklearn, PyTorch, TensorFlow, etc.). This metadata, stored in the MLmodel file, informs MLflow how to load and serve the model later.

A key point is that a logged model can include multiple flavors if you explicitly add them. For example, a scikit-learn model might also be wrapped with a generic python_function flavor so that it can be served using a common interface. However, by default, MLflow logs the model using the specific library’s flavor without creating multiple, separate versions for different frameworks.

In summary:

- **Single Logged Artifact:** MLflow logs one artifact with a flavor that indicates the library used.
- **Flavor Metadata:** The MLmodel file contains flavor information, letting you know how to load or serve the model.
- **Optional Multiple Flavors:** You can add additional flavors manually, but this isn’t done automatically.

This approach keeps the process streamlined and avoids redundant copies, while still enabling flexibility in how the model is later used or deployed.
