Using a surrogate model can be a viable workaround if your production model isn’t directly supported by SHAP. However, there are important considerations:

- **Fidelity:** The surrogate model (e.g., a scikit‑learn tree‑based model) must closely approximate the behavior of your MLlib model. If it differs significantly in predictions or decision boundaries, then the SHAP values from the surrogate might not accurately reflect the true feature contributions of your production model.

- **Compatibility:** Tree-based models in scikit‑learn work very well with SHAP’s TreeExplainer, so if you can train a similar model on the same data, it allows you to get detailed explanations.

- **Validation:** It’s crucial to validate that the surrogate model’s performance is comparable to your MLlib model on key metrics. If they behave similarly, then the explanations from the surrogate can provide valuable insights.

In summary, using a surrogate model can make sense as long as you ensure it reliably mimics your MLlib model. Make sure to compare predictions and validate performance so that the generated SHAP values are meaningful.
