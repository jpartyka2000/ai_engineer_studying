Great question. Let me check the actual benchmark numbers.Here's the interesting thing: **GPT-4o used as VQAScore is actually *better* than CLIP-FlanT5**, not worse. The t2v_metrics repo describes it as achieving "a new state-of-the-art performance" when using GPT-4o for VQAScore.

This makes sense — the original VQAScore paper showed that CLIP-FlanT5 outperformed GPT-4V (the older vision model), but GPT-4o is significantly more capable than GPT-4V was. The VQAScore method itself is model-agnostic — it just computes P("Yes" | image, "Does this figure show {text}?") — so a stronger underlying VQA model produces better scores.

So the tradeoffs are:

| | CLIP-FlanT5-xl (local) | GPT-4o VQAScore |
|---|---|---|
| **Quality** | Very good | Better (SOTA) |
| **Cost** | Free | ~$0.01-0.03 per image |
| **Speed** | Seconds per image (after warmup) | ~1-2s per API call |
| **Setup** | The nightmare we just lived through | Just an API key |
| **Token limit** | 2048 | 128K |
| **Offline** | Yes | No |

For your use case of scoring a handful of FLUX.2 album artwork outputs per prompt, GPT-4o VQAScore would cost pennies and give you the best possible scores. The main reason to keep local is if you want to avoid API dependency or plan to score hundreds/thousands of images.
