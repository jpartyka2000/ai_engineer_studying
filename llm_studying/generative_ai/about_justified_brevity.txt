## Justified Brevity

**Justified brevity** means being **concise for a good reason** - keeping text short when the context, audience, or purpose makes brevity the right choice.

## Examples:

### Justified Brevity ✓
```python
# Emergency alert
"Fire in building. Exit immediately via stairs."
# Short because urgency matters

# API error message
"Error 404: Resource not found"
# Short because developers need quick, scannable info

# Executive summary
"Recommendation: Acquire. ROI: 47%. Risk: Low."
# Short because executives want bottom line
```

### Unjustified Brevity ✗
```python
# Medical instructions
"Take 2."  # Missing critical info: 2 what? When? With food?

# Bug report
"It's broken."  # Too brief - needs reproduction steps

# Tutorial
"Just use Docker."  # Unhelpfully short for beginners
```

## In LLM Evaluation Context:

When evaluating LLM responses, **justified brevity** checks if the model was appropriately concise:

```python
# User: "Is Python interpreted or compiled?"
# LLM (justified brevity): "Python is interpreted."
# ✓ Simple question gets simple answer

# User: "Explain how Python is interpreted"
# LLM (unjustified brevity): "Python is interpreted."
# ✗ User asked for explanation, too brief

# User: "Quick yes/no - can Python use multiple cores?"
# LLM (justified brevity): "Yes, through multiprocessing."
# ✓ Respects "quick yes/no" but adds minimal context
```

## Key Principle:

Brevity is **justified** when:
- User asks for quick/brief answer
- Context demands speed (emergency, quick reference)
- Additional detail would reduce clarity
- Audience prefers conciseness

**Bottom line**: It's about matching response length to actual need - being short when appropriate, not just lazy.
