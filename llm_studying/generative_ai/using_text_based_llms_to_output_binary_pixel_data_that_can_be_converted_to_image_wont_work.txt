Technically yes, you could ask a text LLM to output something like raw pixel values, SVG code, or base64-encoded image data. But the practical results vary a lot depending on the format.

**What actually works reasonably well:**

SVG is probably your best bet. Text LLMs can generate SVG markup that renders as vector graphics — simple shapes, icons, diagrams, and basic illustrations. The LLM understands the structure of SVG and can place rectangles, circles, paths, and text with specific colors and coordinates. For simple, geometric visuals, this can be surprisingly decent.

HTML/CSS canvas or even ASCII art are also viable for simple visuals.

**What doesn't work well:**

Asking an LLM to produce raw binary pixel data (like a bitmap) or base64-encoded image bytes for a photorealistic scene would give you essentially garbage. The model has no spatial reasoning at the pixel level — it's predicting tokens, not composing visual information. It doesn't understand how thousands of RGB values combine to form coherent shapes, lighting, or texture. You'd get noise or at best very crude patterns.

**Why the gap exists:**

Image generation models like FLUX or Stable Diffusion are trained specifically on image-text pairs with architectures designed for spatial coherence (diffusion transformers, latent spaces, etc.). A text LLM is optimized for sequential token prediction. It can describe an image beautifully but can't *render* one at the pixel level.

**My honest opinion on accuracy:**

For simple SVG-style graphics (logos, charts, geometric art, UI mockups), maybe 6/10 — usable with some iteration. For anything photorealistic or complex via raw binary output, essentially 0/10. You'd be far better off using the text LLM to generate a detailed prompt and then passing that prompt to an image model.
