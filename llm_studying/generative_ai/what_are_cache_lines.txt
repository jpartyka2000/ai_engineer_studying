A **cache line** is the smallest fixed-size block of memory that a CPU moves between **main memory (RAM)** and its **cache**.

---

## 1. Why It Exists

* CPUs are **much faster** than RAM. To avoid waiting on RAM all the time, CPUs keep a small, very fast memory called *cache*.
* But instead of fetching one byte at a time, the CPU fetches memory in **chunks** (cache lines).

---

## 2. Typical Size

* On modern CPUs, a cache line is usually **64 bytes**.
* That means when the CPU needs any single byte at address `X`, it pulls in the **entire 64-byte block** around it into cache.

---

## 3. How It Works

* Suppose you access variable `a` at address `0x1000`.
* The CPU loads the cache line covering addresses `0x1000 – 0x103F`.
* If you then access `b` at `0x1008`, it’s already in cache → very fast.
* If you access `c` at `0x2000`, that’s outside the line → a new line is fetched from RAM.

---

## 4. Implications

* **Spatial locality:** Programs run faster if they access data close together in memory, since they’ll likely be in the same cache line.
* **False sharing:** If two CPU threads update different variables that happen to sit in the same cache line, they can slow each other down (because the whole line has to be kept consistent across cores).
* **Alignment matters:** Data structures aligned to cache-line boundaries can reduce waste and avoid false sharing.

---

## 5. Connection to ML / Batch Sizes

When batch sizes align with cache-line sizes (or GPU memory transaction sizes), memory fetches are more efficient. That’s one reason why powers of 2 (or at least multiples of 8/16/32) are common choices in deep learning.

---

✅ **In short:**
A **cache line** is a 64-byte (typically) block that the CPU uses as its unit of caching. Programs that work with data in contiguous memory take advantage of cache lines to run much faster.
