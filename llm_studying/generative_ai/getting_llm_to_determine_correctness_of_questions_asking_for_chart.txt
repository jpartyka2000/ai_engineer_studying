This is a great question. You have a few options, and the right choice depends on what aspects of correctness you care about most.

**Option 1: Judge the underlying data, not the chart**

If the LLM generates code to produce the chart, you can extract or intercept the data that gets plotted. For a line chart, this might be a simple table like:

```
month,organic,paid,referral
2024-09,45.20,62.30,38.10
2024-10,47.50,58.90,40.20
...
```

This is the cleanest approach because it separates "did you get the right data?" from "did you render it correctly?" You can judge data correctness precisely, and rendering correctness is usually less ambiguous.

**Option 2: Extract chart metadata programmatically**

If you have access to the chart object (e.g., a matplotlib figure, Plotly JSON, or a Vega-Lite spec), you can extract structured information: axis labels, series names, data points, chart type, etc. This gives you a rich text representation without relying on vision.

**Option 3: Use a vision model**

Modern multimodal models can look at the rendered chart image directly. You'd provide the image along with the original question and ask the model to assess whether the chart answers the question correctly. This catches visual/rendering issues but can be less precise about exact values.

**Option 4: Hybrid approach**

Judge the data/code for numerical correctness, and optionally use vision to check that the chart type and labels are sensible.

**My recommendation:** If you can access the underlying data or the chart specification (like a Plotly JSON or the dataframe being plotted), go with that. It's more reliable and deterministic than vision, and correctness for charts is mostly about "did you query/aggregate the data correctly?" anyway.
