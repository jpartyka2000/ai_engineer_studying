## Policy Violation Evaluation for LLM Training

Your trajectory generation workflow serves as the foundation for evaluating whether an LLM follows optimal decision-making policies based on complexity heuristics.

### Trajectory Generation (Current Phase)
Generate "oracle" trajectories that define ground truth based on your complexity rules:
```python
{
    "expression": "(2+2)",
    "complexity": 5,
    "complexity_label": "Simple",
    "optimal_action": "verify",  # Heuristic says: try cheap verify first
    "optimal_cost": 1            # Best case if verify succeeds
}
```

### Policy Violations to Detect
- **Type 1**: Using expensive `compute` on Simple expressions (wasteful)
- **Type 2**: Using `verify` on Advanced expressions that will likely fail (inefficient, ends up costing 6)
- The violation is about choosing suboptimal strategies, not mathematical correctness

### Evaluation Process (Next Phase)
1. **Generate trajectories first** as your answer key
2. **Test base LLM** on same expressions to get its decisions
3. **Compare** LLM choices against optimal actions from trajectories
4. **Measure** policy adherence rate, cost efficiency, and violation patterns

### Key Metrics
- Policy adherence: % matching heuristic decisions
- Cost delta: actual vs optimal cost per expression
- Violation distribution: which complexity levels fail most

The trajectories you're generating now become both training data for fine-tuning and the benchmark for measuring whether the base model (and later fine-tuned model) learned to follow your complexity-based decision policy. Generate trajectories first, then use them to evaluate LLM performance.
