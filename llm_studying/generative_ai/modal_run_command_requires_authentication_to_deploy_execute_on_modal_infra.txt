Yes, it makes sense to log into Modal first if you're using the default execution mode.

Looking at your script:

- **Default mode is `modal`** (`EXEC_ENV="${EXEC_ENV:-modal}"`)
- When running in modal mode, the script executes `modal run --detach cue_eval/modal_eval_harness.py::app.run -- "${ARGS[@]}"`

The `modal run` command requires authentication to deploy and execute code on Modal's infrastructure, so you'll need valid credentials before running the script.

**When you need to log in first:**
```bash
# Default behavior (uses Modal)
./cue_eval/run_eval_harness.sh [RUN_NAME] [CONFIG_PATH]

# Explicit Modal mode
EXEC_ENV=modal ./cue_eval/run_eval_harness.sh [RUN_NAME] [CONFIG_PATH]
```

**When you don't need Modal auth:**
```bash
# Local mode bypasses Modal entirely
./cue_eval/run_eval_harness.sh --local [RUN_NAME] [CONFIG_PATH]

# Or via environment variable
EXEC_ENV=local ./cue_eval/run_eval_harness.sh [RUN_NAME] [CONFIG_PATH]
```

So if you intend to run on Modal's cloud infrastructure (which appears to be the primary use case given it's the default), run `modal setup` first to authenticate.
