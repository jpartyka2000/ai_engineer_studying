PATHLIB DEEP DIVE: walk_up PARAMETER IN .relative_to()
═══════════════════════════════════════════════════════════════

AVAILABILITY:
───────────────────────────────────────────────────────────────
Python 3.12+ only (released October 2023)

BASIC SYNTAX:
───────────────────────────────────────────────────────────────
Path.relative_to(other, walk_up=False)  # Default behavior
Path.relative_to(other, walk_up=True)   # New in 3.12

WHAT PROBLEM DOES walk_up SOLVE?
───────────────────────────────────────────────────────────────
Before Python 3.12, .relative_to() could ONLY compute relative
paths when the "other" path was a PARENT (ancestor) of the 
original path. It couldn't handle sibling directories or paths
requiring ".." traversal.

from pathlib import Path

# WORKS: destination is inside start (child relationship)
Path("/home/user/projects/app/main.py").relative_to("/home/user")
# PosixPath('projects/app/main.py')

# FAILS: destination is NOT inside start (sibling relationship)
Path("/home/user/documents").relative_to("/home/user/projects")
# ValueError: '/home/user/documents' is not in the subpath of 
# '/home/user/projects'

This was a significant limitation! To get from /projects to
/documents, you need "../../documents" — but relative_to()
couldn't compute that.

THE SOLUTION: walk_up=True
───────────────────────────────────────────────────────────────
With walk_up=True, the method can now "walk up" the directory
tree using ".." to find a common ancestor, then walk back down.

from pathlib import Path  # Python 3.12+

Path("/home/user/documents").relative_to(
    "/home/user/projects", 
    walk_up=True
)
# PosixPath('../documents')

Path("/home/user/documents/report.pdf").relative_to(
    "/home/user/projects/app",
    walk_up=True
)
# PosixPath('../../documents/report.pdf')

HOW IT WORKS CONCEPTUALLY:
───────────────────────────────────────────────────────────────
Given:
  start: /home/user/projects/app
  dest:  /home/user/documents/report.pdf

Step 1: Find common ancestor
  /home/user/projects/app
  /home/user/documents/report.pdf
  ──────────────────────
  Common ancestor: /home/user

Step 2: Count levels UP from start to common ancestor
  /home/user/projects/app → /home/user
  Need to go up 2 levels: "../.."

Step 3: Path DOWN from common ancestor to destination
  /home/user → /home/user/documents/report.pdf
  Path down: "documents/report.pdf"

Step 4: Combine
  "../.." + "documents/report.pdf" = "../../documents/report.pdf"

COMPARISON: walk_up=False vs walk_up=True
───────────────────────────────────────────────────────────────
from pathlib import Path

dest = Path("/home/user/docs/file.txt")
start_parent = Path("/home/user")
start_sibling = Path("/home/user/projects")

# PARENT RELATIONSHIP — both work the same
dest.relative_to(start_parent, walk_up=False)  # 'docs/file.txt'
dest.relative_to(start_parent, walk_up=True)   # 'docs/file.txt'

# SIBLING RELATIONSHIP — only walk_up=True works
dest.relative_to(start_sibling, walk_up=False) # ValueError!
dest.relative_to(start_sibling, walk_up=True)  # '../docs/file.txt'

MORE EXAMPLES:
───────────────────────────────────────────────────────────────
from pathlib import Path

# Same directory
Path("/home/user/file1.txt").relative_to(
    "/home/user/file2.txt", walk_up=True
)
# PosixPath('../file1.txt')

# Cousin directories (both need up + down)
Path("/var/log/app.log").relative_to(
    "/home/user/projects", walk_up=True
)
# PosixPath('../../../var/log/app.log')

# Deep nesting
Path("/a/b/c/d/e").relative_to("/a/b/x/y/z", walk_up=True)
# PosixPath('../../../c/d/e')

# Completely different trees (still works!)
Path("/mnt/backup/data").relative_to("/home/user", walk_up=True)
# PosixPath('../../mnt/backup/data')

# Root to somewhere
Path("/home/user").relative_to("/", walk_up=True)
# PosixPath('home/user')

# Somewhere to root
Path("/").relative_to("/home/user", walk_up=True)
# PosixPath('../..')

EDGE CASES AND LIMITATIONS:
───────────────────────────────────────────────────────────────
# 1. Different drives on Windows — still fails!
Path("D:/data").relative_to("C:/users", walk_up=True)
# ValueError: 'D:\\data' and 'C:\\users' have different anchors

# 2. Mixing absolute and relative — fails
Path("/home/user").relative_to("relative/path", walk_up=True)
# ValueError: '/home/user' and 'relative/path' have different anchors

# 3. Both must be same type (both absolute or both relative)
Path("a/b/c").relative_to("a/x/y", walk_up=True)
# PosixPath('../../b/c')  # Works! Both relative.

CHECKING PYTHON VERSION:
───────────────────────────────────────────────────────────────
import sys

def relative_path_between(start, dest):
    """Cross-version relative path calculation."""
    start = Path(start)
    dest = Path(dest)
    
    if sys.version_info >= (3, 12):
        return dest.relative_to(start, walk_up=True)
    else:
        import os
        return Path(os.path.relpath(dest, start))

# Usage
relative_path_between("/home/user/projects", "/home/user/docs")
# Works on any Python version!

PRACTICAL USE CASES:
───────────────────────────────────────────────────────────────
# Use Case 1: Creating relative links between project files
def create_relative_link(from_file, to_file):
    """Generate a relative path for linking files."""
    from_path = Path(from_file).resolve()
    to_path = Path(to_file).resolve()
    return to_path.relative_to(from_path.parent, walk_up=True)

# In /project/docs/readme.md, link to /project/src/main.py
create_relative_link(
    "/project/docs/readme.md",
    "/project/src/main.py"
)
# PosixPath('../src/main.py')

# Use Case 2: Build system — express output relative to source
source = Path("/project/src/module/file.py")
output = Path("/project/build/module/file.pyc")
relative = output.relative_to(source.parent, walk_up=True)
# PosixPath('../../build/module/file.pyc')

# Use Case 3: Generate import paths
def python_relative_import(from_module, to_module):
    """Calculate relative import path between modules."""
    from_path = Path(from_module)
    to_path = Path(to_module)
    rel = to_path.relative_to(from_path.parent, walk_up=True)
    
    # Convert to Python import syntax
    parts = rel.with_suffix('').parts
    dots = ''.join('.' for p in parts if p == '..')
    module = '.'.join(p for p in parts if p != '..')
    return f"from {dots} import {module}"

python_relative_import(
    "mypackage/utils/helpers.py",
    "mypackage/core/engine.py"
)
# "from .. import core.engine"

WHY IS walk_up=False THE DEFAULT?
───────────────────────────────────────────────────────────────
Backward compatibility. Existing code relies on .relative_to()
raising ValueError when paths don't have a parent-child 
relationship. Changing the default would break that code.

Also, walk_up=True is slightly more expensive computationally,
and many use cases only need the simpler parent-child check.

COMPARISON WITH os.path.relpath():
───────────────────────────────────────────────────────────────
Feature              | os.path.relpath | relative_to(walk_up=True)
---------------------|-----------------|---------------------------
Returns              | str             | Path object
Handles ".." paths   | Yes             | Yes
Cross-drive (Win)    | Returns abs path| Raises ValueError
Both must be abs/rel | No              | Yes
Available since      | Python 2.6      | Python 3.12

os.path.relpath() is more forgiving but returns a string.
relative_to(walk_up=True) is stricter but returns a Path.

SUMMARY:
───────────────────────────────────────────────────────────────
- walk_up=True allows relative_to() to compute paths requiring
  ".." traversal (sibling/cousin directories)
- Only available in Python 3.12+
- Default is walk_up=False for backward compatibility
- Both paths must have the same anchor (both absolute or both
  relative, same drive on Windows)
- For older Python, use os.path.relpath() as a fallback
