A true “sloping” pattern only exists if you can impose a meaningful order on the categories. In other words:

* **Ordinal (or numeric) features** naturally lend themselves to sloping charts: you plot bins in increasing order (e.g. GPA ranges, age groups) and you expect the target rate to rise/fall monotonically.

* **Nominal (purely string) features** have no intrinsic order. If you simply alphabetize or leave them unsorted, there is no “slope” to speak of. That said, you can still force a “sloping” look by sorting categories according to some metric—for example:

  1. **By frequency** (largest bucket on the left, smallest on the right), or
  2. **By target‐rate** (highest one‐to‐zero ratio on the left, lowest on the right).

  In that case, you’ve created an artificial ordering based on your data, but it isn’t a true ordinal scale—it’s just a plotting convenience to highlight which categories drive the target most.

So if your string column really does represent an ordinal concept (e.g. “Small”, “Medium”, “Large” or “Low”, “Medium”, “High”), then you should map it to 0/1/2 (or define an explicit ordering) before making a sloping plot. If it’s purely nominal (e.g. “Lead\_Source” or “Degree”), the only way to “slope” it is to sort its bars by some summary statistic. That can be useful for visualization, but remember you’re imposing an order that doesn’t exist in the raw data.
