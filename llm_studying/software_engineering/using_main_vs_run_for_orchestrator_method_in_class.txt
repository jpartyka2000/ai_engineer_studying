There are indeed different conventions, and the choice often depends on the context and type of class you're building:

## **`run()` Convention**
Used when the class represents something that **executes** or **operates**:
- **Thread/Process classes**: `Thread.run()`, `Process.run()`
- **Command/Task objects**: Classes that encapsulate a specific operation
- **Agents/Workers**: `Agent.run()`, `Worker.run()`
- **Applications with lifecycle**: `App.run()`, `Server.run()`

```python
class DataPipeline:
    def run(self):  # This pipeline "runs"
        self.extract()
        self.transform()
        self.load()

class Agent:
    def run(self):  # This agent "runs" to complete its task
        self.initialize()
        self.process()
        self.cleanup()
```

## **`main()` Convention**
Used for **entry points** and **top-level orchestration**:
- **Script-like classes**: When the class is essentially a script wrapped in a class
- **Application entry points**: The primary starting point
- **CLI tools**: Often paired with `if __name__ == "__main__"`

```python
class Application:
    def main(self):  # Primary entry point
        self.parse_args()
        self.setup()
        self.execute()

class ScriptRunner:
    @staticmethod
    def main():  # Static method as entry point
        runner = ScriptRunner()
        runner.process_all()
```

## **Other Common Conventions**

- **`execute()`**: When the class represents a command or action
- **`start()`**: For services or long-running processes
- **`process()`**: For data processing classes
- **`__call__()`**: When you want the instance to be callable like a function

```python
class Command:
    def execute(self):  # Command pattern
        pass

class Service:
    def start(self):  # Service pattern
        pass

class Processor:
    def __call__(self, data):  # Callable instance
        return self.process(data)
```

**Rule of thumb**: 
- Use `run()` when your class represents something that actively does work (agents, pipelines, tasks)
- Use `main()` for top-level orchestration or when mimicking a script's entry point
- Consider domain-specific names (`process()`, `analyze()`, `generate()`) when they better describe what the method does

The key is consistency within your codebase and clarity about what the method does.
