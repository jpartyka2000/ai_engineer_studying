### **What is a Mixin?**  
A **mixin** is a **special type of class** in object-oriented programming that is **not meant to be instantiated on its own** but instead provides additional functionality when **combined with other classes** via multiple inheritance.  

Mixins are used to **add reusable behavior** to multiple classes **without** duplicating code.  

---

### **Key Characteristics of a Mixin**
✅ **Cannot be used alone** – It must be mixed into another class.  
✅ **Adds functionality** – It extends behavior **without** affecting the base class.  
✅ **Encourages code reuse** – Instead of duplicating methods, you can create a mixin.  
✅ **Uses multiple inheritance** – Mixins are typically used alongside other classes.  

---

### **Example: Logging Mixin**
Let’s say multiple classes need logging functionality. Instead of duplicating the logging code, we create a **`LoggingMixin`**.

```python
class LoggingMixin:
    def log(self, message):
        print(f"[LOG]: {message}")

class User(LoggingMixin):
    def __init__(self, name):
        self.name = name

    def greet(self):
        self.log(f"User {self.name} has logged in.")
        print(f"Hello, {self.name}!")

# Using the mixin
user = User("Alice")
user.greet()
```
**Output:**
```
[LOG]: User Alice has logged in.
Hello, Alice!
```
👉 `User` **inherits** from `LoggingMixin`, gaining the `log()` method without modifying `User`.

---

### **Example: Multiple Mixins**
You can combine multiple mixins to add different behaviors.

```python
class LoggingMixin:
    def log(self, message):
        print(f"[LOG]: {message}")

class JSONMixin:
    def to_json(self):
        return self.__dict__

class User(LoggingMixin, JSONMixin):
    def __init__(self, name, age):
        self.name = name
        self.age = age

# Using mixins
user = User("Alice", 30)
user.log("This is a log message.")  # From LoggingMixin
print(user.to_json())  # From JSONMixin
```
**Output:**
```
[LOG]: This is a log message.
{'name': 'Alice', 'age': 30}
```

---

### **When to Use Mixins?**
✅ When **multiple classes** need **shared behavior** (like logging, serialization, authentication).  
✅ When you want to **avoid deep inheritance hierarchies** (favoring composition).  
✅ When you want to **add behavior without modifying base classes**.
