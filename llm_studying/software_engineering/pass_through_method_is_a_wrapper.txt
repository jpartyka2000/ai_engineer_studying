A **pass-through method** (also called a "wrapper" or "delegate" method) is a method that doesn't do any real work itself — it simply forwards the call to another object's method and returns the result.

For example, in the `DeltaTableEDA` class:

```python
def remove_html_tags(
    self,
    column: str,
    preserve_spacing: bool = True
) -> pd.DataFrame:
    # This method doesn't contain any logic for removing HTML tags.
    # It just passes the arguments to _advanced_analytics and returns the result.
    result_df = self._advanced_analytics.remove_html_tags(
        self.df, column, preserve_spacing
    )
    self.df = result_df
    return result_df
```

**Why use pass-through methods?**

1. **Convenience**: Users of `DeltaTableEDA` can call `eda.remove_html_tags(column)` instead of `eda._advanced_analytics.remove_html_tags(eda.df, column)`. The interface is simpler.

2. **Encapsulation**: The user doesn't need to know that `AdvancedAnalytics` exists internally. They just interact with `DeltaTableEDA`.

3. **State management**: The pass-through can do small housekeeping tasks, like updating `self.df` with the result, which the underlying method doesn't know about.

4. **Consistent API**: All functionality is accessible through one class rather than requiring users to juggle multiple objects.

The trade-off is that you end up writing thin methods that mostly just call other methods, which can feel repetitive. But it keeps your public API clean.
