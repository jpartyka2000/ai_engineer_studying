## Reasons to Have Both YAML and TOML:

### **Common Valid Reasons:**

### 1. **Different Tools Require Different Formats**
```python
my-project/
├── pyproject.toml       # Python packaging (required)
├── .github/
│   └── workflows/
│       └── ci.yml       # GitHub Actions (YAML only)
├── docker-compose.yml   # Docker (YAML only)
└── mkdocs.yml          # Documentation tool (YAML only)
```

### 2. **Migration in Progress**
```python
my-project/
├── config.yaml         # Old configuration (still in use)
├── config.toml         # New configuration (migrating to)
└── settings.py         # Loads both during transition
```

### 3. **Different Purposes**
```python
my-project/
├── pyproject.toml      # Build/dependency config (static)
└── config.yaml         # Runtime application config (complex nested structure)
```

## **Usually NOT Ideal:**

Having both for the **same purpose** is confusing:
```python
# ❌ Avoid this:
my-project/
├── settings.toml
└── settings.yaml    # Same config in two formats?
```

## **Real-World Example:**
```python
# Common Python project
my-project/
├── pyproject.toml          # Required for Python packaging
├── .pre-commit-config.yaml # Pre-commit hooks (YAML only)
├── kubernetes/
│   └── deployment.yaml     # K8s manifests (YAML only)
└── ansible/
    └── playbook.yml        # Ansible (YAML only)
```

**Bottom line**: Having both is fine when **different tools demand different formats**. Avoid duplicating the same configuration in both formats - pick one for each specific use case.
