This is a practical concern, but it's worth separating the legal question from the social/reputational one.

**Legally**, you're in a very safe position. Maintaining a profanity filter list for content moderation is a well-established, legitimate engineering practice. There's no legal liability for having a list of blocked words in your codebase — you're using it to *prevent* offensive content, not produce it. Companies like Google, Meta, and countless open-source projects maintain similar lists publicly. For example, there are widely-used open-source profanity lists on GitHub (like `profanity-check`, `better_profanity`, etc.) that have thousands of stars and no legal issues.

**Regarding the social/reputational concern**, if someone discovers a blocklist and reacts negatively, that's a PR issue, not a legal one. In practice, the opposite is more likely — people would react negatively if you *didn't* have a filter.

**Practical approaches if you're still concerned:**

You don't need to hardcode the list at all. You could store it in an external config file or environment variable, load it from a remote endpoint at runtime, hash the words (so the raw text isn't readable in source), or use an existing third-party profanity detection library so the list isn't yours to maintain.

A simple comment in the code like `# Blocklist used for content safety filtering` is more than sufficient context. You don't need a legal disclaimer for this. If you're still uneasy, making the repo private is fine too, but it's really not necessary from a legal standpoint.

That said, I'm not a lawyer — if you want formal legal assurance, consulting one would be the way to go.
