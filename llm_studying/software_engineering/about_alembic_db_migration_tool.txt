**Alembic** is a lightweight database migration tool for **SQLAlchemy**-powered applications. It helps you evolve your database schema over time—adding tables, columns, indices, constraints, and more—while preserving existing data.

---

## 1. Why use Alembic?

* **Schema version control**: Just like you manage source code, you manage a linear history of “migration” scripts that describe how to move your schema forward (and backward).
* **Repeatable deployments**: Automate schema changes across development, staging, and production environments.
* **Data preservation**: Migrations run in-place, altering tables rather than dropping and recreating, so you keep your data intact.

---

## 2. Key Concepts

| Concept            | Description                                                                                   |
| ------------------ | --------------------------------------------------------------------------------------------- |
| **Revision**       | A single migration script (Python file) with `upgrade()` and `downgrade()` functions.         |
| **Head**           | The latest revision in your migration history.                                                |
| **Stamp**          | Marking the database with a particular revision without running migrations.                   |
| **Autogeneration** | Alembic can compare your SQLAlchemy models against the database and generate a revision stub. |
| **env.py**         | The project’s migration environment file—sets up the database connection and logging.         |
| **alembic.ini**    | The config file pointing to your `env.py`, database URL, and script directory.                |

---

## 3. Installation

```bash
pip install alembic
```

Then initialize a migrations directory in your project:

```bash
alembic init alembic
```

This creates:

* `alembic/`: folder containing `env.py`, `script.py.mako`, and a `versions/` subfolder for your revisions.
* `alembic.ini`: configuration file.

---

## 4. Basic Workflow

1. **Configure**

   * Edit `alembic.ini` → set `sqlalchemy.url` to your database.
   * Edit `alembic/env.py` → import your SQLAlchemy `Base.metadata` so Alembic knows your models.

2. **Create a revision**

   ```bash
   alembic revision -m "create users table"
   ```

   This makes a new file in `alembic/versions/` with empty `upgrade()`/`downgrade()` stubs.

3. **Edit the revision**

   ```python
   def upgrade():
       op.create_table(
           'user',
           sa.Column('id', sa.Integer(), primary_key=True),
           sa.Column('name', sa.String(50), nullable=False),
       )

   def downgrade():
       op.drop_table('user')
   ```

4. **Apply migrations**

   ```bash
   alembic upgrade head
   ```

   Runs all pending `upgrade()` functions up to the latest revision.

5. **Roll back** (if needed)

   ```bash
   alembic downgrade -1
   ```

   Runs the previous revision’s `downgrade()`. You can also target specific revision IDs.

---

## 5. Autogeneration

Alembic can auto-detect changes in your SQLAlchemy models:

```bash
alembic revision --autogenerate -m "add email to user"
```

It diffs `Base.metadata` vs. the current database schema and writes `op.add_column()` or `op.drop_column()` calls for you. **Always review** autogenerated scripts before applying.

---

## 6. Common Commands

| Command                              | Purpose                                  |
| ------------------------------------ | ---------------------------------------- |
| `alembic init <dir>`                 | Initialize migration environment         |
| `alembic revision -m "<msg>"`        | Create empty new migration               |
| `alembic revision --autogenerate -m` | Create migration by diffing models       |
| `alembic upgrade <rev>`              | Apply migrations up to given revision    |
| `alembic downgrade <rev>`            | Revert migrations down to given revision |
| `alembic history`                    | Show all revisions in order              |
| `alembic current`                    | Show current revision in database        |
| `alembic stamp <rev>`                | Mark DB as at given revision (no ops)    |

---

## 7. Best Practices

* **Keep revisions small & focused**: Easier to review and troubleshoot.
* **Review autogenerated scripts**: Autogen can’t handle complex operations (data migrations, renames) without human tweaks.
* **Use branches sparingly**: Linear history avoids merge conflicts in migration scripts.
* **Test migrations**: Run `upgrade`/`downgrade` in a throwaway database to ensure they work both ways.
* **Source control**: Commit both your models and migration scripts together.

---

**In a nutshell**, Alembic brings the power of version control to your database schema, integrating seamlessly with SQLAlchemy and giving you confidence that schema changes stay in sync across all your environments.
