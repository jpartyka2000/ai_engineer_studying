
Yes—it’s generally seen as a bit of a hack or “kludge.” Monkey‑patching lets you bend objects at runtime in ways they weren’t designed for, which can:

* **Break encapsulation**: you’re reaching into internal attributes (`_model_impl`) that aren’t part of the public API.
* **Be brittle**: if MLflow’s internals change in a patch release, your monkey‑patch could silently stop working.
* **Hinder readability**: other developers won’t expect methods to appear out of thin air, making the code harder to understand and maintain.

Monkey‑patching can be a pragmatic stop‑gap when there’s no alternative, but as soon as you can, it’s better to:

1. Use the framework‑specific loader (`mlflow.sklearn.load_model`, `mlflow.lightgbm.load_model`, etc.) so you get `predict_proba()` natively.
2. Or wrap your model in a proper `pyfunc.PythonModel` at logging time so the pyfunc API itself exposes the methods you need.

In short: monkey‑patching works in a pinch, but it’s best treated as a last resort.
