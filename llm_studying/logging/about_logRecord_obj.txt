A **`LogRecord`** is the core object that the logging system uses to represent ÔÇťone log event.ÔÇŁ Whenever you call a logger method like `logger.info("ÔÇŽ")` or `logger.error("ÔÇŽ")`, under the hood Python creates a `LogRecord` with all the contextual data youÔÇÖll need downstream in handlers and formatters. HereÔÇÖs what you need to know:

---

## Creation & Lifecycle

1. **Generated by**

   * The `Logger.makeRecord()` factory is called inside methods like `Logger.info()`, `Logger.error()`, etc.
   * You almost never instantiate `LogRecord` yourself unless youÔÇÖre building very custom logging behavior.

2. **Passed to**

   * All attached **Filters** first (they can drop or mutate the record).
   * Then to each **Handler**, which formats and emits it (or skips it based on level/filter).

---

## Key Attributes

| Attribute                 | Description                                                            |
| ------------------------- | ---------------------------------------------------------------------- |
| `name`                    | Name of the logger that created it (e.g. `"myapp.module"`)             |
| `levelno` / `levelname`   | Numeric level (e.g. `20`) and textual name (`"INFO"`) of this event    |
| `pathname` / `filename`   | Full path and base filename of the source file where the call was made |
| `lineno`                  | Line number in the source file                                         |
| `funcName`                | Name of the function or method containing the logging call             |
| `msg`                     | The userÔÇÖs message format string (before interpolation)                |
| `args`                    | Arguments tuple passed for message formatting                          |
| `message`                 | The fully formatted message (`msg % args`)                             |
| `created`                 | Time when the record was created (seconds since the epoch, `float`)    |
| `msecs`                   | Millisecond portion of `created`                                       |
| `relativeCreated`         | Milliseconds since the logging module was loaded                       |
| `thread` / `threadName`   | Thread identifier and name                                             |
| `process` / `processName` | Process ID and name                                                    |
| `exc_info`                | Exception tuple if `logger.exception()` was used                       |
| `stack_info`              | Stack traceback text if `stack_info=True` was passed                   |

---

## How Formatters Use It

A `Formatter` typically uses format strings like:

```python
"%(asctime)s %(levelname)s %(name)s:%(lineno)d: %(message)s"
```

Each `%(ÔÇŽ)s` or `%(ÔÇŽ)d` is looked up as an attribute on the `LogRecord`. ThatÔÇÖs why `LogRecord` carries so much metadataÔÇöso you can include whatever context you need.

---

## Customizing & Extending

* **Subclassing**: If you need extra fields (e.g. user ID, request ID), you can subclass `LogRecord` and override `Logger.makeRecord()` to inject your data.
* **Adding attributes at runtime**: Filters can add attributes to the record (e.g. `record.request_id = get_correlation_id()`), and then you can reference `%(request_id)s` in your format.

---

### TL;DR

A `LogRecord` bundles all the information about a single logging eventÔÇöwhere it came from, at what level, with what message, and in which contextÔÇöso that handlers and formatters can produce rich, informative log outputs.
