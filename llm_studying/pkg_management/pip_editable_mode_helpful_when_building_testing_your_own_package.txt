Great question. The key is distinguishing between two different scenarios:

**Using someone else's package:** Yes, you want it installed normally in `site-packages`. You're not modifying the code, you just want a stable version that works.

**Developing your own package:** This is where editable mode shines. Imagine you're building a library and you want to test it as you go. Your workflow might look like:

1. Write some code
2. Test it (maybe run a script that imports your package, or run unit tests)
3. Find a bug or want to add something
4. Edit the code
5. Test again
6. Repeat

Without editable mode, you'd need to run `pip install .` between steps 4 and 5 every single time. That gets tedious fast, especially if you're making lots of small changes.

With editable mode, you install once and then just edit and test freely.

**Another common scenario:** You're working on an application that depends on a library you're also developing. You want your application to use your local in-progress version of the library, not a published version. Editable mode lets you do that while still having the library be a proper importable package.

So it's not about what's "better" in general — it's about whether you're a consumer of the package (normal install) or actively developing it (editable install).
