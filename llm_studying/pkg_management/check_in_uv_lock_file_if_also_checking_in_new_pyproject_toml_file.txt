Yes, you should commit your `uv.lock` file along with the updated `pyproject.toml`.

**Why commit the lock file:**

1. **Reproducibility** — Everyone on your team (and CI) gets the exact same versions of every dependency, including transitive ones
2. **Deterministic builds** — Without the lock file, `uv` would resolve dependencies fresh each time, potentially getting different versions
3. **Security** — You can audit and track exactly when dependency versions changed

**Typical workflow:**

```bash
# After updating pyproject.toml
uv lock              # Regenerate the lock file
uv sync              # Install the new dependencies locally

# Then commit both
git add pyproject.toml uv.lock
git commit -m "Add new dependencies for feature X"
```

**The one exception:** If you're building a *library* (meant to be imported by other projects) rather than an *application*, some teams choose not to commit the lock file since downstream users will have their own dependency resolution. But even then, many still commit it for reproducible development and CI environments.

Since you mentioned you have new code in your branch that needs these dependencies, your teammates won't be able to run that code properly without the updated lock file — they'd get dependency mismatches or missing packages.
