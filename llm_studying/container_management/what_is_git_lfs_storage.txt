Git LFS (Large File Storage) is an extension to Git that handles large files differently than regular Git.

## The problem it solves

Regular Git stores every version of every file directly in the repository. This works great for code (small text files), but becomes a nightmare for large files like:

- Model weights (often gigabytes)
- Images, videos, audio
- Datasets
- Binaries

If you committed a 10GB model file and then updated it 5 times, your repository would balloon to 50GB+, and cloning would take forever.

## How Git LFS works

Instead of storing the actual large file in the Git repository, LFS stores a small pointer file (just a few bytes) that contains:

```
version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 10737418240
```

The actual large file lives on a separate LFS server. When you clone or pull, Git LFS fetches the real files from that server as needed.

## Why Hugging Face used it

Hugging Face Hub is built on Git — every model repository is a Git repo. This gives you versioning, branching, and familiar workflows. Git LFS was the natural choice for handling the large weight files.

## Why they're moving toward Xet

Git LFS has limitations at Hugging Face's scale — millions of models, petabytes of weights, massive concurrent downloads. Xet is their next-generation solution designed specifically for this use case, with better deduplication, chunking, and transfer speeds.
